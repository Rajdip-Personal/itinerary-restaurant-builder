# Writing Good Requirements

This guide defines the standards for writing requirements in the Agentic AI Workshop. Every requirement generated by agents or written by engineers must meet these quality criteria.

---

## Characteristics of Good Requirements

### 1. Specific
A requirement must be precise enough that two different engineers would interpret it the same way.

**Bad:** The system should be fast.
**Good:** API response time for search queries must be ≤ 500ms at the 95th percentile under normal load (100 concurrent users).

### 2. Testable
Every requirement must have acceptance criteria that can be verified through testing (automated or manual).

**Bad:** The system should be secure.
**Good:** All API endpoints must return 401 Unauthorized when called without a valid authentication token, verified by integration tests.

### 3. Traceable
Every requirement must trace back to a business need (PRD section) and forward to implementation (user stories).

**Bad:** Add caching.
**Good:** FR-015: Implement response caching for the compliance dashboard API (traces to PRD Section 5.2, Goal: sub-second dashboard load times).

### 4. Independent
Requirements should be implementable independently of each other where possible. When dependencies exist, they must be explicitly stated.

**Bad:** Build the reporting module.
**Good:** FR-020: Generate daily compliance summary report. Dependencies: FR-010 (data collection), TR-005 (database schema).

### 5. Prioritized
Every requirement must have a priority that reflects its business importance and implementation order.

| Priority | Meaning | Guideline |
|----------|---------|-----------|
| **P0 — Must Have** | System doesn't work without it | Core functionality, security, compliance |
| **P1 — Should Have** | Significant value, but workarounds exist | Important features, performance optimization |
| **P2 — Nice to Have** | Enhances experience, not critical | UI polish, advanced reporting, automation |

## Requirement Templates

### Business Requirement (BR-XXX)

```markdown
### BR-001: [Business Outcome Title]

- **Description:** [What business outcome this enables. Focus on the "why" — the business value, not the technical implementation.]
- **Priority:** P0 | P1 | P2
- **Acceptance Criteria:** [How we measure that this business outcome has been achieved. Use quantifiable metrics where possible.]
- **Dependencies:** [Other requirements this depends on, if any.]
- **Stakeholder:** [Who owns this requirement — product owner, business sponsor, etc.]
- **PRD Reference:** [Section of the PRD this comes from.]
```

**Example:**
```markdown
### BR-001: Track Employee RTO Compliance

- **Description:** Enable leadership to track which employees are meeting return-to-office requirements, with visibility into compliance rates by team, department, and organization.
- **Priority:** P0
- **Acceptance Criteria:** Leadership can view a dashboard showing daily RTO compliance rates broken down by org level, with data no more than 24 hours old.
- **Dependencies:** None
- **Stakeholder:** VP of Supply Chain Operations
- **PRD Reference:** Section 1 — Product Overview
```

### Technical Requirement (TR-XXX)

```markdown
### TR-001: [Technical Constraint/Standard Title]

- **Description:** [Technical constraint or standard the system must meet. Focus on the "what" — the technical requirement, not the implementation details.]
- **Priority:** P0 | P1 | P2
- **Acceptance Criteria:** [How we verify this technical requirement is met. Should be automatable where possible.]
- **Dependencies:** [Other requirements this depends on.]
- **Standard Reference:** [Link to Nordstrom engineering standard, if applicable.]
```

**Example:**
```markdown
### TR-001: PostgreSQL Database on Standard Platform

- **Description:** Application data must be stored in PostgreSQL hosted on the standard Nordstrom database platform, with automated backups and point-in-time recovery.
- **Priority:** P0
- **Acceptance Criteria:** Database provisioned on standard platform, automated daily backups verified, point-in-time recovery tested to within 5 minutes.
- **Dependencies:** None
- **Standard Reference:** Nordstrom Data Platform Standards
```

### Functional Requirement (FR-XXX)

```markdown
### FR-001: [System Capability Title]

- **Description:** [What the system must do. Focus on behavior — what happens when a user or system takes an action.]
- **Priority:** P0 | P1 | P2
- **Acceptance Criteria:**
  - Given [precondition], When [action], Then [expected result]
  - Given [another precondition], When [another action], Then [another result]
- **Dependencies:** [Other requirements this depends on.]
- **User Persona:** [Which user type this serves.]
```

**Example:**
```markdown
### FR-001: Submit Daily RTO Status

- **Description:** Employees must be able to submit their daily RTO status indicating whether they were in the office, on PTO, working from home, or out sick.
- **Priority:** P0
- **Acceptance Criteria:**
  - Given an authenticated employee, When they submit their status for today selecting "In Office," Then the system records the submission with timestamp and the employee sees a confirmation.
  - Given an authenticated employee, When they submit their status for a past date, Then the system rejects the submission with a message "Status can only be submitted for the current day."
  - Given an authenticated employee who has already submitted today, When they submit again, Then the system updates their existing submission (not creates a duplicate).
- **Dependencies:** TR-001 (database), NFR-001 (authentication)
- **User Persona:** Employee
```

### Non-Functional Requirement (NFR-XXX)

```markdown
### NFR-001: [Quality Attribute Title]

- **Description:** [Quality attribute the system must exhibit. Focus on measurable characteristics.]
- **Priority:** P0 | P1 | P2
- **Acceptance Criteria:** [Measurable threshold or verifiable condition. Must include specific numbers or conditions.]
- **Dependencies:** [Other requirements this depends on.]
- **Category:** Security | Performance | Reliability | Observability | Compliance
```

**Example:**
```markdown
### NFR-001: API Response Latency

- **Description:** All user-facing API endpoints must respond within acceptable latency thresholds under normal and peak load conditions.
- **Priority:** P0
- **Acceptance Criteria:**
  - p50 response time ≤ 200ms under normal load (100 concurrent users)
  - p95 response time ≤ 500ms under normal load
  - p99 response time ≤ 2000ms under peak load (500 concurrent users)
  - Verified by load testing in staging environment before production deployment
- **Dependencies:** TR-001 (database), TR-003 (caching)
- **Category:** Performance
```

## Acceptance Criteria Patterns

### Given/When/Then (Preferred)
Use for behavioral requirements — describes how the system behaves in specific scenarios.

```gherkin
Given [a specific precondition or context]
When [a specific action is taken]
Then [a specific, observable outcome occurs]
```

**Rules:**
- **Given** must set up a specific, reproducible state (not vague like "a user").
- **When** must describe a single action (not multiple actions chained together).
- **Then** must describe an observable, verifiable outcome (not an internal implementation detail).

### Threshold-Based (For NFRs)
Use for quality attributes that have measurable thresholds.

```markdown
- [Metric] must be [comparison] [threshold] under [condition]
- Verified by [measurement method]
```

**Example:**
```markdown
- Error rate must be ≤ 0.1% under normal load (100 concurrent users)
- Verified by 30-minute load test in staging environment with production-like data
```

### Checklist-Based (For Compliance)
Use for requirements that have multiple verifiable conditions.

```markdown
- [ ] Condition 1 met
- [ ] Condition 2 met
- [ ] Condition 3 met
```

**Example:**
```markdown
- [ ] All API endpoints return 401 without valid auth token
- [ ] RBAC roles enforced at middleware layer
- [ ] Authorization decisions logged with user ID and resource
- [ ] Session expires after 30 minutes of inactivity
```

## Common Mistakes

| Mistake | Why It's Bad | Fix |
|---------|-------------|-----|
| "The system should be user-friendly" | Not testable or specific | Define specific usability criteria (task completion time, error rate) |
| "Support high availability" | No measurable target | "Availability ≥ 99.9% measured monthly" |
| "Handle errors gracefully" | Vague — what does "gracefully" mean? | "Return structured error response with error code, message, and correlation ID" |
| "Must be scalable" | No specific scale target | "Support 1000 concurrent users with p95 latency ≤ 500ms" |
| "Use best practices" | Not verifiable | List the specific practices required |
| Combining multiple requirements | Hard to trace, test, and prioritize independently | Split into separate requirements with unique IDs |
| No acceptance criteria | Can't verify if requirement is met | Always include at least one Given/When/Then or measurable threshold |
